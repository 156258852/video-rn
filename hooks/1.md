我们重点关注你提到的 bug：**拖动进度条跨 clip 时，目标 clip 总是从 0 秒开始播放，而不是预期的拖动位置。**

---

## 1. 根源分析：seek 在 `SEEK_TO_LOAD` 路径中没有得到保证

跨 clip 拖动最终会调用 `seekToClip`，它分两种情况：

- 如果目标 clip **已经加载** 在另一个 slot 中 → dispatch `SWITCH_TO_CLIP`，并在 dispatch 前直接调用 `playerRefs[nextPlayer].current.seek(time)` ✅ 这里 seek 是可靠的。
- 如果目标 clip **尚未加载** → dispatch `SEEK_TO_LOAD`，将期望的 seek 信息放入 `pendingAction`，然后 **完全依赖** `onLoad` 回调中的逻辑去执行 seek。

`onLoad` 中的逻辑大致为：

```ts
const onLoad = (e) => {
  const s0 = stateRef.current;
  // ...
  if (s0.pendingAction && s0.pendingAction.clipIdx === videoIndex) {
    playerRefs[i].current?.seek?.(s0.pendingAction.time);
  } else if (s0.activePlayer !== i) {
    playerRefs[i].current?.seek?.(0);  // 预加载槽 seek(0)
  }
  dispatch({ type: 'LOAD_COMPLETE', ... });
};
```

**问题就出在这个依赖链上：**

- `SEEK_TO_LOAD` 会立即切换 `activePlayer` 并改变 `currentIndex`，这会引发 `videoSlots` 重新计算。
- 如果目标 clip 原本已经被**预加载槽**开始加载（只是尚未完成 `onLoad`），那么 `SEEK_TO_LOAD` 之后，该 slot 的 source **没有变化**（URI 相同），React Native Video 组件**不会重新挂载，也不会再次触发 `onLoad`**。
- 但此时已经存在的加载过程结束时，`onLoad` 确实会被调用 —— 这时 `stateRef.current.pendingAction` 已经是新的 seek 请求，理论上应该能进入 `if` 分支去 seek。

**但仍然会在以下情况中失败：**

### ① 在 `onLoad` 调用时，`playerRefs[i].current.seek` 尚未就绪

- 在 React Native Video 某些版本 / 平台上，`onLoad` 回调触发时，native 侧的 seek 方法可能还不能立刻接受指令，导致 `seek(time)` 被无声地忽略。
- 此时视频会停在默认位置（通常为 0），第一个 `onProgress` 会携带 `currentTime = 0`，而 reducer 在 `Loading → Playing` 转换时会把 `times[clipIdx]` 覆盖为这个 0，最终表现为**从 0 开始播放**。

### ② 预加载槽的 source 相同导致 `onLoad` 不被调用的另一种变体

- 假设目标 clip 是 `currentIndex + 1`，且该槽之前已经**成功加载过**该 clip（例如用户从更后面的位置拖回来），那么 `slotAssignments` 中已有该 clip 的记录，此时 `seekToClip` 会走 `SWITCH_TO_CLIP` ✅，正常工作。
- 如果该槽**曾经加载过该 clip**，但后来因为 `RESET` 或其它原因 `slotAssignments` 被清空，而 Video 实例的 source 并未实际改变（因为 URI 没变）——这时 `SEEK_TO_LOAD` 被触发，组件不重新挂载，`onLoad` 不会执行，**seek 完全丢失**。

这两种典型情况都会导致“拖动跨 clip 后从 0 开始播放”。

## 2. 状态流转中的其他隐患（与 bug 相关）

### 2.1 在 `Loading` 状态下收到 `PROGRESS` 会直接覆盖 `times`

```ts
if (state.phase === 'Loading') {
  return {
    ...state,
    phase: 'Playing',
    times: updateTimes(state.times, state.times.length, event.clipIdx, t),
    currentTime: t,
    // ...
  };
}
```

- 如果 seek 没有成功，第一个 progress 带 `t=0`，就会把之前 `SEEK_TO_LOAD` 已经写入的 `times[clipIdx] = 5` 重置为 0。
- 虽然这反映了真实播放位置，但用户期望的是 seek 到 5，不应该让进度条跳回 0。

### 2.2 `pendingAction` 在 `LOAD_COMPLETE` 中被立即清空

```ts
if (state.pendingAction && state.pendingAction.clipIdx === event.clipIdx) {
  return {...state, slotAssignments: newSlots, pendingAction: null};
}
```

- 这里假设 seek 已经在外部的 `onLoad` 中完成，因此直接清除 `pendingAction`。
- 但如果 seek 实际失败了，这个清除动作就使得后续无法重试 seek。

---

## 3. 修复建议

**核心思路：不再仅依赖 `onLoad` 中的 seek，而是使用一个 `useEffect` 来响应 `pendingAction`，确保 seek 被可靠执行。**

### 3.1 新增一个 effect，监听 `pendingAction` 并执行 seek

```ts
useEffect(() => {
  const action = state.pendingAction;
  if (!action) return;
  // 只处理 'seek' 类型，'resume' 有自己的逻辑
  if (action.type !== 'seek') return;

  const slot = state.activePlayer; // pendingAction 总是针对 active slot
  const player = playerRefs[slot].current;

  if (player && typeof player.seek === 'function') {
    // 立即尝试 seek，如果失败可以稍后重试（可选）
    player.seek(action.time);
  }
}, [state.pendingAction, state.activePlayer, playerRefs]);
```

### 3.2 修改 `onLoad`，去除其中的主动 seek 逻辑，仅保留状态同步

- 删除 `onLoad` 中针对 `pendingAction` 的 `seek` 调用，只做 `LOAD_COMPLETE` dispatch。
- 对于预加载槽，仍然保留 `seek(0)` 以保证起始位置正确（或者也可由 effect 统一处理）。

### 3.3 增加重试保护（可选）

- 如果担心 `player.seek` 在 ref 刚绑定时仍无效，可以在 effect 中加入 `requestAnimationFrame` 或短 `setTimeout` 延迟执行。

## 4. 修改后的关键代码片段

```tsx
// 新增 effect
useEffect(() => {
  const action = state.pendingAction;
  if (!action || action.type !== 'seek') return;

  const slot = state.activePlayer;
  const player = playerRefs[slot].current;
  if (player && player.seek) {
    player.seek(action.time);
  }
}, [state.pendingAction, state.activePlayer]);

// 在 videoSlots 的 onLoad 中，移除 seek 调用，只保留：
const onLoad = (e: any) => {
  const s0 = stateRef.current;
  const dur = Number(e?.duration);
  if (recordDurationRef.current && Number.isFinite(dur)) {
    recordDurationRef.current(videoIndex, dur);
  }

  // 预加载槽 seek 到 0（避免从旧位置开始）
  if (s0.activePlayer !== i) {
    playerRefs[i].current?.seek?.(0);
  }
  // 不再主动 seek pendingAction.time，交给 useEffect 统一处理

  dispatch({
    type: 'LOAD_COMPLETE',
    slotIdx: i,
    clipIdx: videoIndex,
    duration: dur,
  });
};
```

这样一来，**无论组件是否重新挂载，只要 `pendingAction` 存在，effect 都会确保 seek 执行**，从而修复“拖动跨 clip 后总是从 0 播放”的问题。
